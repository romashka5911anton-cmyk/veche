# Вклад в ВЕЧЕ

Спасибо, что хотите помочь. ВЕЧЕ специально написан просто — чтобы порог входа был низким. Не нужно знать фреймворки: только базовый JavaScript.

## С чего начать

```bash
git clone https://github.com/romashka5911anton-cmyk/veche.git
cd veche
npm install
npm start        # http://localhost:4040
```

Всё. Никакой сборки: правите `public/app.js` или `public/style.css` — обновляете вкладку. Правите `server.js` — перезапускаете `npm start`.

Прочитайте [ARCHITECTURE.md](ARCHITECTURE.md) — там модель данных, события сокета и точки расширения.

## Стиль кода

- **Чистый JS, без сборщиков и фреймворков.** Это принцип, а не временное решение — пусть код останется читаемым для любого.
- Пишите в том же стиле, что и рядом: те же имена, та же плотность комментариев. Комментарии — по-русски, по делу.
- Зависимости добавляйте только если без них правда никак. Сейчас их три — берегите это.
- Одна фича/фикс — один осмысленный PR. Опишите, **что** и **зачем**.

## Криптография: правьте с особой осторожностью

Файл `public/crypto.js` — единственное место, где живёт шифрование секретных чатов. Ошибка здесь не падает с исключением: она просто тихо оставляет людей без защиты, на которую они рассчитывали. Поэтому правила жёстче обычных.

- **Никакой самописной криптографии.** Сейчас используется только штатный Web Crypto (`window.crypto.subtle`): ECDH P-256 + HKDF-SHA256 + AES-256-GCM. Своих реализаций примитивов, своих режимов, своих KDF в проекте нет и не должно быть.
- **Не меняйте схему без обсуждения.** Смена кривой, шифра, порядка вывода ключей, формата шифроблока — сначала issue с обоснованием, потом код. Учтите совместимость: у людей уже лежат ключи в IndexedDB, и старую переписку надо уметь читать.
- **Приватный ключ не должен становиться извлекаемым и не должен покидать браузер.** Он создаётся с `extractable=false` и хранится в IndexedDB; на сервер уходит только публичный. Правку, которая это нарушает, не примем.
- **Не ослабляйте текущие свойства:** одноразовая (эфемерная) пара ключей на каждое сообщение, ключ сообщения заворачивается отдельно для каждого получателя, сервер по e2e-сообщению хранит пустой `text` и непрозрачный блок шифра, ИИ и модерация по нему не работают, в push уходит «🔒 Зашифрованное сообщение».
- **Требуется HTTPS.** Web Crypto доступен только в secure context — локально проверяйте с этим в уме.
- **В PR пишите честно, что именно проверили.** «Кажется, работает» здесь не аргумент.

### Что не сделано — можно взяться

Ниже честный список того, чего в шифровании **нет**. Это открытые задачи, а не скрытые недостатки:

- 🔑 **Прямая секретность (PFS) и ratchet.** Сейчас их нет: компрометация приватного ключа получателя раскрывает всю прошлую переписку.
- 📌 **Пиннинг ключа.** Публичный ключ собеседника берётся из данных сервера, предупреждения о смене ключа нет. Единственная защита — ручное сравнение «кода сверки» (SHA-256 от пары публичных ключей, 15 групп по 5 цифр). Значит, вредоносный сервер теоретически может попытаться подменить ключ.
- ✍️ **Подписи сообщений.** Сообщения не подписываются: авторство утверждает только серверное поле «от кого».
- 📎 **Шифрование вложений.** Фото, файлы, голосовые и кружки не шифруются — в секретном режиме они просто заблокированы в интерфейсе.
- 👥 **Групповое E2E.** Группы и каналы шифрования не имеют. (Чат с ИИ-ассистентом сквозным быть не может в принципе.)
- 🔄 **Мультиустройство.** Ключ привязан к одному браузеру: вход с нового устройства создаёт новый ключ и перезаписывает публичный на сервере, старая секретная переписка там не расшифруется. Очистка данных браузера = безвозвратная потеря ключа и доступа к своей секретной переписке.
- 🧪 **Автотесты крипты.** Их нет вовсе — хорошая и полезная первая задача для этого раздела.

Два ограничения снять патчем нельзя, и о них надо помнить: клиентский JS отдаёт тот же сервер (принципиальная слабость любого веб-E2E), а метаданные — кто, кому, когда, факт переписки, прочтения, реакции — сервер видит всегда. Секретный режим при этом остаётся локальным флагом устройства: он не синхронизируется и не мешает собеседнику писать в тот же чат открытым текстом.

Независимого профессионального аудита криптографии не было. Если вы умеете такое — это самый ценный вклад, который здесь можно сделать.

## Прежде чем прислать PR

- Проверьте синтаксис: `node -c server.js`.
- Прогоните вручную ключевые сценарии: регистрация, отправка сообщения, создание группы, звонок. Опишите в PR, что проверили.
- Не коммитьте `data.json`, `uploads/`, `.env` — они в `.gitignore`.
