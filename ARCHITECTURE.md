# Как устроено ВЕЧЕ

Документ для тех, кто хочет разобраться в коде и дополнить его. Здесь — модель данных, протокол событий, ключевые потоки и точки расширения.

Философия проекта: **минимум магии**. Один файл сервера, клиент без сборки, три зависимости. Всё можно прочитать глазами за вечер.

---

## Обзор

```
                    Браузер / PWA
              ┌───────────────────────┐
              │  index.html            │
              │  app.js  (весь клиент) │
              │  crypto.js (E2E, лички)│
              │  style.css · sw.js     │
              └───────────┬───────────┘
                   HTTP   │   WebSocket (Socket.IO)
                          ▼
              ┌───────────────────────┐
              │      server.js         │
              │  ┌─────────────────┐   │
              │  │ Express (HTTP)  │   │  /api/*  + статика + /uploads
              │  ├─────────────────┤   │
              │  │ Socket.IO       │   │  чаты, звонки, игры, лента…
              │  ├─────────────────┤   │
              │  │ db (в памяти)   │   │  → data.json (отложенная запись)
              │  └─────────────────┘   │
              │  lib/games.js          │
              │  lib/moderation.js     │
              └───────────┬───────────┘
                          ▼
                 data.json  +  uploads/
```

Один процесс Node держит всё состояние в оперативной памяти (объект `db`) и лениво сбрасывает его в `data.json`. Файлы (фото, голос, кружки) лежат на диске в `uploads/` и раздаются статикой.

`public/crypto.js` — отдельный маленький модуль клиента (`VecheCrypto`), вся криптография секретных чатов 1-на-1. Работает только по HTTPS (нужен secure context), использует исключительно стандартный Web Crypto (`window.crypto.subtle`) — самописной криптографии в проекте нет. Содержимого таких сообщений сервер не видит.

## Модель данных

Всё хранится в объекте `db` (см. верх `server.js`):

```js
db = {
  users:     [],   // аккаунты
  messages:  [],   // ВСЕ сообщения всех чатов (плоский массив)
  chats:     [],   // группы и каналы (личные чаты и «Эфир» — виртуальные)
  scheduled: [],   // отложенные сообщения (ждут своего времени)
  posts:     [],   // лента / стена
  reports:   [],   // жалобы для модерации
  reminders: [],   // напоминания от ассистента
  pushSubs:  {},   // userId -> [Web Push подписки]
  vapid:     null, // ключи VAPID (генерируются один раз)
}
```

### User
```js
{
  id, username, displayName, bio, avatar, photos: [],
  tg, phone, email,
  salt, passHash,               // scrypt(пароль, salt)
  token,                        // токен сессии (Bearer)
  pubKey,                       // публичный ключ E2E (ECDH P-256, base64); приватный сервер не видит никогда
  colors: [c1, c2],             // персональный градиент аватара
  createdAt, lastSeen,
  privacy:  { closed, showPhone, ghost },
  settings: { muteNewChats, hideOnline, hideEfir },
  contacts: [], blocked: [], muted: [],
  admin,   banned,  isBot,  customStatus,
}
```

### Message
```js
{
  id, chatId, from,             // from=null для системных сообщений
  type,                         // text|image|voice|file|geo|circle|video|album|poll|game|system|e2e
  text, media,                  // форма media зависит от type (см. doSend)
                                // для type='e2e': text всегда '', шифр лежит в media.enc
  ts, readBy: [],  reactions: { '👍': [userId, …] },
  edited, deleted, pinned,
  replyTo, forward, ttl,        // ttl — секунды до самоуничтожения
  scheduledFor,                 // если отложено
}
```

### Chat (только группы и каналы)
```js
{
  id,                           // 'grp:…' или 'chn:…'
  type,                         // 'group' | 'channel'
  title, description, avatar,
  owner, admins: [], members: [], subscribers: [],   // subscribers — читатели канала
  createdAt,
}
```

### Идентификаторы чатов

| Вид | Формат `chatId` | Пример |
|-----|-----------------|--------|
| Эфир | константа `'efir'` | `efir` |
| Личка | `dm:` + два id через `:` (отсортированы) | `dm:ab12:cd34` |
| Группа | `grp:` + случайный id | `grp:9f8e…` |
| Канал | `chn:` + случайный id | `chn:1a2b…` |

Личные чаты и «Эфир» **не хранятся** как объекты — они выводятся из сообщений и участников. Это упрощает модель: `dmId(a, b)` детерминированно даёт один и тот же id для пары.

## Хранилище

- Всё состояние — в памяти. `save()` дебаунсит запись `data.json` на 250 мс, чтобы серия событий не била по диску.
- Обычные сообщения хранятся открытым текстом: сервер их видит, и они лежат в `data.json` как есть. Исключение — сообщения типа `e2e` (секретные чаты 1-на-1): у них `text = ''`, а в `media.enc` лежит непрозрачный для сервера блок шифра, расшифровать который сервер не может.
- Плюсы: нулевая настройка, атомарная логика, легко бэкапить (`cp data.json backup.json`) и читать глазами.
- Минусы: масштаб ограничен ОЗУ и одним процессом. Массив `messages` линейно растёт.

**Когда упрётесь в лимиты** — это первое, что стоит заменить. Точки замены сосредоточены: `save()`, начальная загрузка, и функции доступа (`findUser`, `findChat`, фильтры по `db.messages`). Разумный следующий шаг — SQLite (таблицы users / chats / messages, индекс по `chatId, ts`) или Postgres. Socket-слой при этом не меняется.

## HTTP API

Аутентификация — заголовок `Authorization: Bearer <token>`.

| Метод | Путь | Назначение |
|------|------|-----------|
| POST | `/api/register` | регистрация (первый человек → админ) |
| POST | `/api/login` | вход, выдаёт новый токен |
| POST | `/api/profile` | обновление профиля/приватности/настроек, а также публикация `pubKey` (публичного E2E-ключа) |
| POST | `/api/password` | смена пароля |
| POST | `/api/upload` | загрузка бинарника (фото/голос/файл), `body` = raw |
| GET  | `/api/link-preview?url=` | Open Graph превью (с защитой от SSRF) |
| GET  | `/api/push/keys` | публичный VAPID-ключ |
| POST | `/api/push/subscribe` · `/unsubscribe` | подписка на пуши |
| GET  | `/api/call/pending` · POST `/api/call/decline` | входящий звонок, если приложение было закрыто |
| POST | `/api/mobile-log` | «чёрный ящик» нативного клиента |

Загрузки раздаются из `/uploads` с `X-Content-Type-Options: nosniff`; всё, кроме безопасного медиа, отдаётся как вложение (защита от stored XSS).

## Протокол Socket.IO

При коннекте сокет аутентифицируется по токену (`io.use(...)`), затем получает событие **`init`** со снимком состояния: `{ me, users, chats, iceServers }`.

**Клиент → сервер** (основное):

| Событие | Действие |
|---------|----------|
| `history` `{chatId}` | запросить последние сообщения чата |
| `send` `{chatId, type, text, media, replyTo, forward, ttl, scheduledFor}` | отправить сообщение |
| `edit` · `delete` · `react` · `pin` · `unpin` | операции над сообщением |
| `read` `{chatId}` · `typing` · `erasing` | статусы |
| `mute` `{chatId, on}` | заглушить чат |
| `createChat` · `updateChat` · `addMembers` · `removeMember` · `setAdmin` · `leaveChat` | группы/каналы |
| `call:offer` · `answer` · `ice` · `decline` · `end` · `cancel` | сигналинг WebRTC |
| `game:create` · `game:move` | мини-игры |
| `vote` `{id, option}` | голос в опросе |
| `createPost` · `feed` · `wall` · `likePost` · `deletePost` | лента/стена |
| `report` · `reportMsg` · `getReports` · `banUser` · `dismissReport` · `deleteAnyMsg` | жалобы и модерация |

**Сервер → клиент** (основное): `message`, `edited`, `deleted`, `reacted`, `voted`, `pinned`, `read`, `typing`, `erasing`, `presence`, `user_joined`, `user_updated`, `chat_created`, `chat_info`, `chat_removed`, `game_update`, `new_post`, `post_liked`, `post_deleted`, `call:*`, `new_report`, `banned_notice`.

Доставка адресная: `emitToChat()` шлёт событие всем участникам чата (по личным «комнатам» `socket.join(userId)`), «Эфир» — всем в комнате `efir`.

## Ключевые потоки

**Отправка сообщения** (`doSend`): валидация типа/медиа → антиспам (`spamCheck`: частота + повторы + `moderate`) → нормализация `chatId` → проверка прав (`canSend`) → приватность (закрытый профиль/блок) → сборка объекта → сохранение → `emitToChat('message')` → побочные эффекты: самоуничтожение (`ttl`), ответ ассистента (`maybeAlisaReply`), пуши офлайн-получателям (`notifyOffline`).

**Секретное сообщение** (`type: 'e2e'`) идёт по тому же `doSend`, но с несколькими отличиями. Сервер ставит `text = ''`, пересобирает `media.enc` только из известных полей с жёсткими лимитами длины, требует непустой `media.enc` (иначе `bad enc`) и **отклоняет e2e вне лички** — если `chatId` не начинается на `dm:`, ответ будет `Секретные сообщения — только в личке`. Ассистент «Алиса» на такие сообщения не вызывается, модерация по тексту невозможна (текста нет), а в push уходит фиксированная строка «🔒 Зашифрованное сообщение».

Поток целиком (см. `public/crypto.js` и `sendSecret` в `app.js`):

1. **Ключ личности.** При первом включении секретного режима клиент генерирует пару ECDH P-256 через `subtle.generateKey`. Приватный ключ создаётся с `extractable = false` и кладётся в IndexedDB браузера как `CryptoKey` — прочитать его байты нельзя даже своему же JS.
2. **Публикация.** На сервер уходит **только публичный** ключ (base64, `POST /api/profile` → поле `pubKey`) и дальше раздаётся другим клиентам в `init` / `user_updated`. Приватный ключ не покидает устройство.
3. **Эфемерная пара на каждое сообщение.** Перед отправкой генерируется ещё одна пара P-256 — одноразовая, только для этого сообщения.
4. **HKDF.** ECDH между эфемерным приватным и публичным ключом получателя даёт общий секрет (256 бит) → `HKDF-SHA256` (salt = сырой эфемерный публичный ключ, info = `veche-e2e-v1`) → ключ-обёртка AES-256-GCM.
5. **AES-GCM.** Отдельно берётся случайный ключ сообщения (32 байта), им шифруется текст (`AES-256-GCM`, случайный 12-байтный IV). Сам ключ сообщения заворачивается ключом-обёрткой **отдельно для каждого получателя** — включая самого отправителя, иначе он не прочитает собственное сообщение.
6. **Сервер.** Получает и хранит объект вида `{ v, ephPub, iv, ct, keys: { userId: {iv, wk} } }`. Ни одного ключа расшифровки в нём нет — для сервера это непрозрачный блоб, который он просто пересылает и складывает в `data.json`.
7. **Расшифровка при рендере.** Получатель при отрисовке сообщения берёт свою запись из `keys`, повторяет ECDH+HKDF уже своим приватным ключом из IndexedDB, разворачивает ключ сообщения и расшифровывает текст. Расшифрованное живёт только в памяти вкладки (поле `_dec`).

**Код сверки** (safety number): `SHA-256` от пары публичных ключей, отформатированный как 15 групп по 5 цифр. Собеседники сравнивают его вручную (голосом, при встрече) — это единственный способ убедиться, что сервер не подсунул чужой ключ.

Честные границы этой схемы (подробнее — в [SECURITY.md](SECURITY.md)):

- Шифруется **только текст**. Вложения (фото, файлы, голосовые, кружки) не шифруются — в секретном режиме они просто заблокированы в интерфейсе.
- Только личка 1-на-1. Группы, каналы и «Эфир» не шифруются; обычные (не секретные) личные чаты тоже хранятся открыто. Чат с ИИ-ассистентом принципиально не может быть E2E.
- Метаданные — кто, кому, когда, факт переписки, прочтения, реакции — сервер видит всегда.
- Нет прямой секретности (PFS) и ratchet: компрометация приватного ключа получателя раскрывает всю прошлую переписку.
- Нет пиннинга ключа: публичный ключ собеседника берётся из данных сервера, предупреждения о смене ключа нет. Значит, вредоносный сервер теоретически может попытаться подменить ключ — защита только ручное сравнение кода сверки.
- Сообщения не подписываются: авторство утверждает лишь серверное поле «от кого».
- Ключ привязан к одному браузеру. Вход с нового устройства создаёт **новый** ключ и перезаписывает публичный на сервере — старая секретная переписка там не расшифруется. **Очистка данных браузера = безвозвратная потеря ключа и доступа к своей секретной переписке.**
- Секретный режим — локальный флаг устройства (`localStorage`): он не синхронизируется и не мешает собеседнику писать в этот же чат открытым текстом.
- Клиентский JS отдаёт тот же сервер — принципиальное ограничение любого веб-E2E.
- Независимого профессионального аудита криптографии не было.

**Звонок** (WebRTC): звонящий шлёт `call:offer` → сервер релеит получателю (и кладёт в `pendingCalls` + пуш, если тот офлайн) → `call:answer` / обмен `call:ice` идут через сервер. Сам звук/медиа — напрямую P2P (или через TURN). Сервер только сводит стороны.

**Режим «Призрак»**: `visibleUsersFor`, `broadcastUserMeta`, `broadcastPresence` фильтруют, кто кого видит. Призрака видят только те, с кем есть общая история/чат/контакт (`knowsEachOther`).

## Точки расширения

**Добавить тип сообщения:** внесите тип в белый список в `doSend`, опишите разбор `media`, добавьте отрисовку в `renderBody()` (клиент, `app.js`) и превью в `previewText()`.

**Добавить игру:** правила — в `lib/games.js`; обработка ходов — в `game:move` (`server.js`); отрисовка — в `buildGame()` (`app.js`).

**Сменить ИИ-провайдера:** «Алиса» изолирована в `server.js` (`yandexGPT`, `maybeAlisaReply`, `speechKitTranscribe`). Замените реализацию `yandexGPT()` на вызов любого другого LLM — остальное не изменится. Без ключей ассистент вежливо сообщает, что не настроен.

**Настроить модерацию:** редактируйте `lib/moderation.js` — это отдельный файл именно для того, чтобы каждая площадка задавала свои правила. Секретные (`e2e`) сообщения модерации недоступны в принципе — сервер не видит их текста.

**Доработать шифрование:** вся клиентская криптография собрана в `public/crypto.js` (`VecheCrypto`: `ensureIdentity`, `encrypt`, `decrypt`, `safetyNumber`). Самые заметные незакрытые задачи — PFS/ratchet, пиннинг ключа с предупреждением о его смене, подпись сообщений, шифрование вложений и перенос ключа между устройствами.

**Заменить хранилище:** см. раздел «Хранилище» выше.

## Известные ограничения (и куда расти)

- **Один процесс.** Нет горизонтального масштабирования из коробки (Socket.IO при кластеризации требует адаптера, напр. Redis). Для сотен-тысяч пользователей на инстанс — ок; для миллионов — нужна другая архитектура.
- **Плоский массив сообщений** линейно растёт; поиск/история — это фильтры по массиву. Первый кандидат на БД.
- **E2E-шифрование есть только в секретных чатах 1-на-1** (включается вручную кнопкой-замком), и шифруется там только текст. Во всём остальном — группы, каналы, «Эфир», обычные лички, вложения — сервер видит содержимое. Метаданные видны всегда, прямой секретности (PFS) нет, аудита криптографии не было. См. [SECURITY.md](SECURITY.md).
- **Нет федерации** — инстансы независимы. См. [FEDERATION.md](FEDERATION.md).
- **`data.json` без ротации/сжатия** — для больших инсталляций делайте бэкапы и планируйте миграцию на БД.

Это не недостатки, а честная граница ответственности «маленького сильного мессенджера». Каждый пункт — приглашение к доработке.
